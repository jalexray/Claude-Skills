# Launch Workstreams

This skill launches Claude sessions in separate git worktrees for parallel execution of workstreams created by `/segment-workstreams`.

## Usage

Invoke with: `/launch-workstreams <sprint-name>`

Or with specific workstreams: `/launch-workstreams <sprint-name> --only infrastructure,feature-a`

---

## Phase 1: Load Workstream Manifest

### Actions

1. **Locate the workstreams manifest** using Glob:
   - Search for `*<sprint-name>*-workstreams.md`

2. **Read the manifest** to extract:
   - List of all workstreams
   - Execution order and dependencies
   - Which workstreams block others

3. **Locate individual workstream files**:
   - Search for `*<sprint-name>*-workstream-*.md`
   - Match each to the manifest

4. **If files not found**: Ask user for correct sprint name or paths

---

## Phase 2: Determine Launch Strategy

### Actions

1. **Check current git state**:
   ```bash
   git status
   git worktree list
   ```

2. **Identify already-running worktrees**:
   - Check if any workstream worktrees already exist
   - Report their status

3. **Determine which workstreams can launch now**:
   - **Ready**: No blockers, can start immediately
   - **Blocked**: Waiting on other workstreams to complete
   - **Running**: Already has an active worktree

4. **Present options to user** using AskUserQuestion:

   ```
   Workstreams for [sprint-name]:

   READY TO LAUNCH:
   - infrastructure (blocks: feature-a, feature-b)
   - feature-c (independent)

   BLOCKED (waiting on infrastructure):
   - feature-a
   - feature-b

   ALREADY RUNNING:
   - (none)

   Which workstreams would you like to launch?
   ```

   Options:
   - Launch all ready workstreams
   - Launch specific workstreams (let user choose)
   - Launch infrastructure only (blocking workstream first)
   - Show dependency graph and exit

---

## Phase 3: Launch Worktrees

For each selected workstream, launch a Claude session in a dedicated worktree.

### Launch Command Format

```bash
claude --worktree <sprint>-<workstream> "<initial-prompt>"
```

### Initial Prompt Template

The initial prompt passed to each worktree session:

```
You are starting work on the [workstream-name] workstream for [sprint-name].

Read the workstream file at [path-to-workstream-file] for full context including:
- Tasks to complete
- Files you'll be modifying
- Dependencies and interface contracts
- Completion criteria

Start by reading that file, then begin working through the tasks in order.
Use /rename "[sprint-name] - [workstream-name]" to name this session.
```

### Launch Modes

#### Mode 1: Sequential Launch (Default for blocking workstreams)

For workstreams that block others (like infrastructure), launch in the current terminal and wait:

```bash
claude --worktree myfeature-infrastructure "Read myfeature-workstream-infrastructure.md and work through the tasks. Use /rename to name this session."
```

After this completes, proceed to launch parallel workstreams.

#### Mode 2: Parallel Launch (For independent workstreams)

For workstreams that can run in parallel, provide commands the user runs in separate terminals:

**Output to user:**
```
## Launch Parallel Workstreams

Open new terminals and run these commands:

### Terminal 1: feature-a
claude --worktree myfeature-feature-a "Read myfeature-workstream-feature-a.md and work through the tasks."

### Terminal 2: feature-b
claude --worktree myfeature-feature-b "Read myfeature-workstream-feature-b.md and work through the tasks."

### Terminal 3: feature-c
claude --worktree myfeature-feature-c "Read myfeature-workstream-feature-c.md and work through the tasks."
```

#### Mode 3: Generate Launcher Script

Create an executable script for convenience:

```bash
#!/bin/bash
# myfeature-launch.sh
# Generated by /launch-workstreams

SPRINT="myfeature"

echo "Launching workstreams for $SPRINT"
echo ""

# Detect terminal emulator and launch accordingly
launch_in_new_terminal() {
    local name=$1
    local cmd=$2

    if [[ "$OSTYPE" == "darwin"* ]]; then
        # macOS - use osascript to open new Terminal tabs
        osascript -e "tell application \"Terminal\" to do script \"$cmd\""
    elif command -v gnome-terminal &> /dev/null; then
        # Linux with GNOME Terminal
        gnome-terminal -- bash -c "$cmd; exec bash"
    elif command -v xterm &> /dev/null; then
        # Fallback to xterm
        xterm -e "$cmd" &
    else
        echo "Could not detect terminal. Run manually:"
        echo "  $cmd"
    fi
}

# Launch parallel workstreams
launch_in_new_terminal "feature-a" "claude --worktree ${SPRINT}-feature-a \"Read ${SPRINT}-workstream-feature-a.md and work through the tasks.\""
launch_in_new_terminal "feature-b" "claude --worktree ${SPRINT}-feature-b \"Read ${SPRINT}-workstream-feature-b.md and work through the tasks.\""
launch_in_new_terminal "feature-c" "claude --worktree ${SPRINT}-feature-c \"Read ${SPRINT}-workstream-feature-c.md and work through the tasks.\""

echo ""
echo "Workstreams launched in new terminals."
echo "Use 'claude --resume' to see all active sessions."
```

### Phase 3 Actions

1. **Ask user for launch mode** using AskUserQuestion:
   - Launch blocking workstream now (interactive)
   - Generate launch commands to copy/paste
   - Generate launcher script
   - Launch with specific workstreams only

2. **For interactive launch**:
   - Use Bash to run `claude --worktree <name> "<prompt>"`
   - This will take over the terminal

3. **For command generation**:
   - Output formatted commands user can copy
   - Include the full initial prompt with workstream file path

4. **For script generation**:
   - Write script to `SPRINTNAME-launch.sh`
   - Make it executable: `chmod +x`
   - Report file location

## Phase 4: Post-Launch Summary

After launching or generating launch commands:

```
## Workstreams Launched

Sprint: [sprint-name]
Launched: [N] workstreams

### Active Sessions

| Workstream | Worktree | Session Name |
|------------|----------|--------------|
| infrastructure | .claude/worktrees/myfeature-infrastructure | myfeature - infrastructure |
| feature-a | .claude/worktrees/myfeature-feature-a | myfeature - feature-a |

### Useful Commands

# List all worktrees
git worktree list

# List all Claude sessions
claude --resume

# Resume a specific session
claude --resume "myfeature - feature-a"

# Check on a worktree's branch
git log worktree-myfeature-feature-a --oneline -5

### When Workstreams Complete

1. In each worktree, commit your changes
2. Merge to main: git checkout main && git merge worktree-<name>
3. Remove worktree: git worktree remove <path>

Or let Claude handle cleanup when you exit each session.
```

---

## Error Handling

- **Manifest not found**: Ask user for correct sprint name or run `/segment-workstreams` first
- **Workstream file missing**: Report which file is missing, ask user to regenerate
- **Worktree already exists**: Report existing worktree, offer to resume that session instead
- **Git state dirty**: Warn user about uncommitted changes in main, ask whether to proceed
- **Branch conflict**: If worktree branch already exists, offer to reuse or create new

## Examples

### Launch infrastructure first, then parallel features

```
/launch-workstreams myfeature
```

Output:
```
Workstreams for myfeature:

READY TO LAUNCH:
- infrastructure (blocks: feature-a, feature-b, feature-c)

BLOCKED:
- feature-a (waiting on: infrastructure)
- feature-b (waiting on: infrastructure)
- feature-c (waiting on: infrastructure)

Recommended: Launch 'infrastructure' first, then parallel workstreams after.

Launching infrastructure now...
```

### Generate launcher script for team

```
/launch-workstreams myfeature --script
```

Output:
```
Generated: myfeature-launch.sh

Run with: ./myfeature-launch.sh

This will open new terminals for each parallel workstream.
```

### Launch specific workstreams only

```
/launch-workstreams myfeature --only feature-a,feature-b
```

Output:
```
Launching 2 workstreams:
- feature-a
- feature-b

### Terminal 1: feature-a
claude --worktree myfeature-feature-a "Read myfeature-workstream-feature-a.md..."

### Terminal 2: feature-b
claude --worktree myfeature-feature-b "Read myfeature-workstream-feature-b.md..."
```
